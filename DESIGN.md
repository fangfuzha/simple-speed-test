# 跨平台测速网页程序详细设计文档

## 1. 背景

当前仓库是一个 Node.js + Express 的最小原型，提供 `/ping`、`/download`、`/upload` 三个接口和一个简单网页。它适合验证思路，但无法满足下面的目标：

- Rust 实现
- 跨平台桌面端托盘快速访问
- Docker 部署
- 自建测速节点与网页界面统一
- 安全限流、防刷、安装/卸载完整清理

本设计文档定义后续要开发的正式版本。

## 2. 产品目标

### 2.1 核心目标

- 提供可本地部署的测速网页。
- 支持浏览器直接访问，也支持桌面托盘快速打开。
- 支持 Docker 一键部署。
- 支持下载、上传、延迟、抖动、丢包测试。
- 面向公网环境提供基础限流与防刷。
- 安装与卸载过程尽量无残留。

### 2.2 非目标

- 不追求与商业测速平台完全一致的全球节点网络。
- 不优先做账号体系、云端同步或复杂社交分享。
- 不将结果永久存储作为默认行为。

## 3. 需求确认与默认决策

以下需求已作为项目固定约束：

- 测速模型：公网测速与自建节点测速都支持。
- 指标范围：下载、上传、延迟、抖动、丢包都支持。
- 测试策略：每次测试 5 秒，并发 5 个，样本次数 5，启用去极值规则。
- 结果统计：最终结果取中位数。
- 结果存储：仅前端临时存储，刷新后丢失。
- 节点能力：测试到当前节点的网络连接。
- 安全要求：对同一公网 IP 做一定程度限流防刷。
- 平台：全平台支持。
- 托盘菜单：打开测速页、设置、退出。
- 设置二级菜单：开机启动、启动时打开浏览器。
- 端口占用策略：支持默认端口与自定义端口；桌面端弹窗提示，Docker 版终端提示。
- 默认开机自启：开启。
- 托盘图标：中英双语与高 DPI 适配。
- 镜像架构：amd64 与 arm64 都支持。
- Docker 编排：提供 compose。
- Docker 网络：默认使用桥接网络。
- 卸载要求：跨平台安装程序卸载时完全清理，尽量避免遗留文件。

## 4. 产品形态

本项目建议拆成两个发行形态，共用同一套核心测速引擎：

### 4.1 桌面端

- Rust 桌面程序。
- 提供系统托盘菜单。
- 内置本地服务与本地网页入口。
- 启动后可自动打开默认浏览器，也可仅驻留托盘。

### 4.2 服务器端

- Rust Web 服务。
- 通过 Docker 运行。
- 对外提供测速网页和测试接口。
- 支持反向代理部署。

## 5. 总体架构

建议采用单仓库、多 crate、共享核心逻辑的架构。

### 5.1 模块划分

- `core`：测速协议、数据模型、统计、限流、配置、节点管理。
- `server`：HTTP API、静态资源托管、Docker 运行入口。
- `desktop`：系统托盘、窗口管理、开机启动、浏览器拉起、安装器集成。
- `web`：前端页面和交互逻辑。

### 5.2 运行模式

1. 桌面模式
   - 启动 Rust 后台服务。
   - 托盘常驻。
   - 本地浏览器打开测速页或由用户手动打开。

2. Docker 模式
   - 仅启动 Web 服务。
   - 通过浏览器访问容器端口。
   - 由 compose 负责持久化配置和端口映射。

## 6. 技术选型

### 6.1 后端

- Rust 2021 或更新版本。
- `tokio` 作为异步运行时。
- `axum` 作为 HTTP 框架。
- `tower` 用于中间件和限流组合。
- `serde` / `serde_json` / `toml` 用于配置和数据序列化。
- `tracing` 用于日志。
- `sqlx` 或 `rusqlite` 用于可选历史扩展。

### 6.2 桌面端

- `tauri` v2 作为跨平台桌面壳。
- 使用系统托盘 API 完成菜单、状态与快捷操作。
- 使用 Tauri 的单实例、自动启动、窗口控制能力。

### 6.3 前端

- 建议采用独立前端工程，构建为静态资源。
- 页面负责发起测速请求、展示过程状态、计算统计结果、临时保存当前会话数据。
- 结果不落盘，不做默认持久化。

### 6.4 容器与发布

- Docker 多阶段构建。
- `amd64` 与 `arm64` 双架构镜像。
- 提供 `docker-compose.yml`。
- 桌面端按平台分别打包安装程序。

## 7. 功能设计

### 7.1 页面功能

- 显示当前节点信息。
- 显示公网或自建节点模式。
- 提供开始测试按钮。
- 逐项显示下载、上传、延迟、抖动、丢包结果。
- 显示测试进度、当前轮次、当前样本。
- 显示限流或防刷拦截提示。

### 7.2 测速模式

#### 7.2.1 公网测速模式

- 面向外部用户访问。
- 默认使用当前部署节点。
- 用于测试用户到当前服务器的网络质量。

#### 7.2.2 自建节点测速模式

- 面向管理员或内网用户。
- 可选择指定节点或本地节点。
- 适合多节点或多站点场景。

### 7.3 指标定义

- 下载速度：客户端向服务器发起并发下载请求，统计有效吞吐。
- 上传速度：客户端上传固定大小数据，统计有效吞吐。
- 延迟：基于轻量 `/ping` 请求测量 RTT。
- 抖动：一组延迟样本的离散程度。
- 丢包：在指定次数的轻量探测中，未成功完成的比例。

## 8. 测试策略设计

### 8.1 固定参数

- 目标总耗时：5 到 10 秒。
- 延迟探测次数：6 次。
- 下载/上传样本次数：3 次。
- 下载/上传并发 worker：3 个。
- 单轮窗口：约 900 毫秒。
- 去极值规则：当样本数达到 5 及以上时，剔除最高值和最低值，再对剩余样本做统计。
- 结果统计：最终结果取中位数。

### 8.2 统计规则

#### 下载和上传

- 每个样本使用 3 个并发 worker。
- 每个样本窗口约 900 毫秒。
- 对每个样本计算 Mbps。
- 结果集合按中位数输出。

#### 延迟和抖动

- 延迟采用多次轻量请求。
- 抖动可用相邻 RTT 差值或标准化离散程度估算。
- 最终值取中位数。

#### 丢包

- 使用固定次数轻量探测。
- 计算失败比例。
- 多轮测试后取中位数。

### 8.3 去极值规则

- 当样本数大于等于 5 时，去掉最高值和最低值。
- 当样本数少于 5 时，不做去极值，直接计算中位数。
- 若样本出现明显异常波动，应在 UI 中标记“结果可信度偏低”。

## 9. 请求与接口设计

### 9.1 基础接口

- `GET /`：测速首页。
- `GET /api/ping`：轻量 RTT 测试。
- `GET /api/download?size=...`：下载测试。
- `POST /api/upload`：上传测试。
- `GET /api/config`：返回前端需要的运行配置。
- `GET /api/health`：健康检查。

### 9.2 配置接口

- `GET /api/settings`：读取当前配置。
- `POST /api/settings`：保存配置，仅桌面端或管理员模式允许。

### 9.3 节点接口

- `GET /api/node`：返回当前节点信息。
- `GET /api/nodes`：多节点场景下返回可用节点列表。

## 10. 前端状态设计

### 10.1 临时存储规则

- 仅保存在内存状态中。
- 刷新页面即丢失。
- 不写 `localStorage`、`IndexedDB` 或磁盘文件作为默认行为。

### 10.2 页面状态

- 当前节点。
- 当前测试模式。
- 当前轮次和样本。
- 当前统计结果。
- 运行中错误和限流提示。

## 11. 节点与测速路由策略

### 11.1 当前节点测速

- 用户访问哪个节点，就默认测哪个节点。
- 节点返回自身标识、版本、地理位置标签、能力标签。

### 11.2 多节点扩展

- 后续可支持节点列表。
- 前端可手动切换节点。
- 默认策略可按 RTT 最低或地理标签选择。

## 12. 安全与限流设计

### 12.1 限流目标

- 防止单个公网 IP 反复刷接口。
- 防止下载、上传接口被滥用。
- 限制高频 `/ping` 请求。

### 12.2 限流策略

- 按公网 IP 做 token bucket 或漏桶限流。
- 对不同接口设不同阈值。
- 对上传和下载请求设置更严格的并发上限。
- 当触发限流时返回明确错误码和前端提示。

### 12.3 额外防护

- 限制单次下载最大体积。
- 限制单次上传最大体积。
- 设置请求超时。
- 在反向代理后保留真实 IP 时，需要严格配置可信代理链。

## 13. 配置设计

### 13.1 配置来源

- 环境变量。
- 配置文件。
- 桌面端设置界面。

### 13.2 配置项

- 监听地址和端口。
- 是否允许自动打开浏览器。
- 是否开机自启。
- 是否启用限流。
- 限流阈值。
- 节点名称。
- UI 语言偏好。
- Docker 场景是否启用 HTTPS 反向代理建议提示。

### 13.3 端口策略

- 默认端口固定。
- 若端口占用，桌面端弹窗提示可改端口。
- Docker 版在终端打印占用说明和下一步操作建议。

## 14. 托盘与桌面端设计

### 14.1 托盘菜单

- 打开测速页。
- 设置。
- 退出。

### 14.2 设置二级菜单

- 开机启动。
- 启动时打开浏览器。

### 14.3 桌面端行为

- 默认开机自启。
- 最小化到托盘而不是强制关闭。
- 关闭窗口时默认隐藏到托盘，真正退出由托盘菜单触发。

### 14.4 图标与语言

- 提供中英文菜单文案。
- 图标提供多尺寸资源以适配高 DPI。
- Windows、macOS、Linux 三端分别验证托盘显示效果。

## 15. Docker 设计

### 15.1 镜像目标

- 支持 `amd64`。
- 支持 `arm64`。

### 15.2 运行方式

- 默认使用桥接网络。
- 提供 `docker-compose.yml`。
- 通过环境变量控制端口、节点名、限流阈值、是否输出详细日志。

### 15.3 挂载与持久化

- 仅在需要保留配置时挂载配置目录。
- 默认不持久化前端结果。
- 若将来启用历史功能，再单独挂载数据卷。

### 15.4 容器提示

- 端口占用或配置缺失时，在终端打印明确提示。
- 提示包含默认访问地址、端口映射和排错建议。

## 16. 安装与卸载设计

### 16.1 桌面端安装包

- Windows：建议 MSI 或 NSIS。
- macOS：建议 `.dmg` 或 `.pkg`。
- Linux：建议 AppImage、deb、rpm 至少覆盖一个主流安装形式。

### 16.2 卸载清理要求

卸载时应尽量清理以下内容：

- 程序主文件。
- 托盘图标缓存。
- 自动启动注册项。
- 配置文件。
- 临时文件。
- 本地日志文件。
- 快捷方式和开始菜单项。

### 16.3 清理边界

- 默认策略应“尽量完全清理”。
- 若用户显式选择保留配置，则仅保留配置目录，不保留运行态缓存。
- 卸载完成后应提示实际删除了哪些类别的文件。

## 17. 目录结构建议

```text
repo/
  crates/
    core/
    server/
    desktop/
  web/
  configs/
  deploy/
    docker/
    compose/
  docs/
    DESIGN.md
```

## 18. 数据生命周期

- 运行态测速结果：只在前端内存中存在。
- 服务端日志：按保留策略轮转。
- 配置：仅保存用户显式修改的设置。
- 卸载：默认清理运行文件与配置文件。

## 19. 错误处理原则

- 端口占用时提供明确的可操作提示。
- 网络超时、后端中断和限流要有不同错误码。
- 测速中断时保留已完成样本，不丢弃前面有效结果。
- 前端展示统一错误态，并提示重试入口。

## 20. 可观测性

- 记录启动、停止、端口绑定、限流触发、测试失败等关键事件。
- 日志默认偏简洁，开发模式可输出调试级别信息。
- Docker 中日志输出到标准输出，便于容器平台采集。

## 21. 验收标准

- 桌面端能通过托盘打开本地测速页。
- Docker 容器可通过浏览器访问测速页。
- 下载、上传、延迟、抖动、丢包可正常测量。
- 多轮测试结果按中位数输出。
- 刷新页面后结果消失。
- 同一公网 IP 的高频请求会触发限流。
- 安装与卸载后不应留下明显运行残留。

## 22. 建议的实现顺序

1. 先实现 `core` 的测速统计和限流逻辑。
2. 再实现 `server` 的 HTTP 接口与静态前端。
3. 然后实现 `desktop` 托盘和自动启动。
4. 最后补 Docker、多架构构建和安装包卸载清理。

## 23. 当前仓库的迁移建议

现有原型可以作为第一阶段验证，但正式版本建议逐步替换为 Rust 实现：

- 保留前端交互思路。
- 将 Express 后端替换为 Rust Web 服务。
- 将简单测速逻辑替换为固定策略、可配置统计和限流中间件。
- 把桌面托盘和安装包作为独立层，而不是塞在 Web 服务里。
